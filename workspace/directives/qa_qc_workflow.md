# 📋 QA/QC 워크플로우 SOP

> 1계층 지침: 코드 품질 보증을 위한 4단계 개발-검토-수정-승인 프로세스.

---

## 목표

모든 코드 변경에 대해 체계적인 품질 보증 프로세스를 적용하여 프로덕션 안정성을 보장한다.

---

## STEP 1 — 개발 노드

### 입력
- `user_request`: 사용자의 작업 요청

### 프롬프트

```
당신은 시니어 소프트웨어 엔지니어입니다.

## 작업 요청
{{user_request}}

## 개발 규칙
- 가독성 높은 코드를 작성하고 함수/변수명은 명확하게 지을 것
- 모든 함수에는 간단한 주석을 포함할 것
- 환경변수, API 키, 시크릿은 절대 하드코딩하지 말 것
- 엣지 케이스(빈 값, null, 예외 상황)를 반드시 고려할 것
- 코드 하단에 유닛 테스트 코드를 함께 작성할 것

## 출력 형식
1. 구현 코드
2. 테스트 코드
3. 사용 방법 (3줄 요약)

작업을 시작하세요.
```

### 출력
- 구현 코드 + 테스트 코드 + 사용법 요약

---

## STEP 2 — QA 자동 검토 노드

### 입력
- `step1_output`: STEP 1의 전체 출력

### 프롬프트

```
당신은 날카로운 QA 엔지니어입니다. 감정 없이 오직 문제점만 지적합니다.

## 검토 대상 코드
{{step1_output}}

## 검토 항목 (각 항목을 반드시 확인)

### [기능성]
- 요구사항을 완전히 충족하는가?
- 누락된 기능이 있는가?
- 입력값 유효성 검사(Validation)가 있는가?

### [보안]
- SQL Injection, XSS, CSRF 취약점이 없는가?
- 민감한 정보(키, 비밀번호)가 코드에 노출되지 않는가?
- 외부 입력값을 신뢰 없이 처리하는 부분이 없는가?

### [안정성]
- 예외 처리(try-catch, error handling)가 충분한가?
- 무한 루프, 메모리 누수 가능성이 없는가?
- 비동기 처리 오류 가능성이 없는가?

### [코드 품질]
- 중복 코드(DRY 원칙 위반)가 없는가?
- 함수가 단일 책임 원칙(SRP)을 지키는가?
- 테스트 코드가 실제로 핵심 로직을 검증하는가?

## 출력 형식

QA 결과: [PASS ✅ / FAIL ❌]

### 발견된 문제 목록
| 번호 | 심각도 (HIGH/MED/LOW) | 문제 내용 | 수정 방향 |
|------|----------------------|-----------|-----------|
| 1    |                      |           |           |

### 총평
(문제 없으면 "이상 없음. STEP 3으로 진행하세요." 출력)
```

### 출력
- QA 결과 (PASS/FAIL) + 문제 목록 + 총평

---

## STEP 3 — 코드 수정 노드

### 입력
- `step1_output`: STEP 1의 원본 코드
- `step2_output`: STEP 2의 QA 결과

### 프롬프트

```
당신은 시니어 소프트웨어 엔지니어입니다. QA 검토 결과를 반영하여 코드를 개선합니다.

## 원본 코드
{{step1_output}}

## QA 지적 사항
{{step2_output}}

## 수정 규칙
- QA에서 지적된 모든 항목을 빠짐없이 수정할 것
- 수정하지 않은 항목이 있다면 반드시 이유를 명시할 것
- 수정으로 인해 기존 기능이 깨지지 않도록 할 것
- 수정된 부분에는 주석으로 # [QA 수정] 태그를 달 것

## 출력 형식
1. 수정 완료된 최종 코드 (전체)
2. 수정 사항 요약 (항목별)
3. 수정하지 못한 항목 및 이유 (있을 경우)
```

### 출력
- 수정된 최종 코드 + 수정 요약 + 미수정 사유

---

## STEP 4 — QC 최종 승인 보고서 노드

### 입력
- `step3_output`: STEP 3의 최종 코드
- `user_request`: 원래 요구사항

### 프롬프트

```
당신은 QC 매니저입니다. 최종 배포 전 공식 QC 보고서를 작성합니다.

## 최종 코드
{{step3_output}}

## 원래 요구사항
{{user_request}}

## QC 보고서 작성 항목

### 1. 요구사항 충족 여부
- 원래 요청한 기능이 모두 구현되었는가? (Yes/No + 근거)

### 2. 최종 체크리스트
- [ ] 코드가 실행 가능한 상태인가?
- [ ] 테스트 코드가 포함되어 있는가?
- [ ] 보안 취약점이 제거되었는가?
- [ ] 환경변수 처리가 올바른가?
- [ ] 예외 처리가 완료되었는가?
- [ ] 주석 및 문서화가 충분한가?

### 3. 예상 리스크
(배포 후 발생 가능한 문제점을 LOW/MED/HIGH로 분류)

### 4. 테스트 방법
(운영 반영 전 확인해야 할 테스트 시나리오 3가지)

### 5. 롤백 계획
(문제 발생 시 원복 방법)

### 6. 최종 판정
배포 승인: [✅ 승인 / ⚠️ 조건부 승인 / ❌ 반려]
판정 이유: 

---
보고서 작성일: (오늘 날짜)
검토 버전: v1.0
```

### 출력
- QC 보고서 (체크리스트, 리스크, 테스트 시나리오, 롤백, 최종 판정)

---

## 마스터 프롬프트 (통합 실행)

노드 분리가 어려울 때 아래 프롬프트 하나로 전체 프로세스를 실행합니다.

```
당신은 개발팀 전체입니다. 아래 요청을 개발 → QA → 수정 → QC 순서로 스스로 처리하세요.

## 작업 요청
{{user_request}}

---

## [PHASE 1] 개발
시니어 엔지니어로서 요청을 구현하세요.
- 주석 포함, 엣지 케이스 처리, 테스트 코드 포함
- 시크릿 하드코딩 금지

---

## [PHASE 2] QA 검토
방금 만든 코드를 냉정한 QA 엔지니어로서 검토하세요.
- 기능성 / 보안 / 안정성 / 코드 품질 4가지 기준
- 문제를 표로 정리 (심각도 포함)

---

## [PHASE 3] 수정
QA에서 발견된 모든 문제를 수정한 최종 코드를 작성하세요.
- 수정 부분에 # [QA 수정] 주석 달기

---

## [PHASE 4] QC 보고서
최종 코드에 대한 QC 보고서를 작성하세요.
- 체크리스트 / 리스크 / 테스트 시나리오 / 롤백 계획 / 최종 판정 포함

---

지금 PHASE 1부터 순서대로 시작하세요.
```

---

## 도구

| 도구 | 용도 |
| ---- | ---- |
| 개발 도구 (Claude Code / Codex) | STEP 1, 3 코드 작성 |
| QA 도구 (Claude Code) | STEP 2 자동 검토 |
| QC 도구 (Claude Code) | STEP 4 보고서 작성 |

## 노드 구성 요약

| 노드명 | 사용 프롬프트 | 입력 변수 | 출력 |
|--------|-------------|-----------|------|
| 개발 | STEP 1 | `{{user_request}}` | 코드 + 테스트 |
| QA 검토 | STEP 2 | `{{step1_output}}` | QA 결과표 |
| 코드 수정 | STEP 3 | `{{step1_output}}`, `{{step2_output}}` | 최종 코드 |
| QC 보고서 | STEP 4 | `{{step3_output}}`, `{{user_request}}` | QC 보고서 |

## 예외 처리

- **STEP 2에서 PASS**: STEP 3 생략 가능, 바로 STEP 4 진행.
- **STEP 4에서 반려**: 반려 사유를 반영하여 STEP 1부터 재시작.
- **조건부 승인**: 조건 항목만 수정 후 STEP 4 재실행.
- **긴급 핫픽스**: STEP 1 + STEP 4만 실행 (QA/수정 생략), 단 보고서에 "긴급 핫픽스" 명시.
