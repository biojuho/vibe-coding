# AI 도구 역량 매트릭스

> 각 도구의 강점을 파악하여 적재적소에 배치합니다.
> 사용자가 태스크를 지시할 때 이 매트릭스를 참고하여 담당 도구를 결정합니다.

## 도구별 프로필

### Claude Code (Opus 4.6)

| 항목 | 내용 |
|------|------|
| 강점 | 대규모 리팩토링, 복잡한 버그 디버깅, 테스트 작성, 코드 리뷰/QA, 아키텍처 설계 |
| 컨텍스트 | CLAUDE.md + `.claude/` memory (대화 간 영속) |
| MCP | Notion, GitHub, SQLite, Telegram, Cloudinary, Brave Search, Filesystem, System Monitor |
| 제약 | 원격 push 불가 (ADR-002), 로컬 실행만 가능 |
| 최적 태스크 | 크리티컬 버그 수정, 멀티파일 리팩토링, 테스트 커버리지 확대, 시스템 감사 |

### Gemini (Antigravity)

| 항목 | 내용 |
|------|------|
| 강점 | 장시간 멀티턴 세션, 반복적 파이프라인 작업, QC 검증, 인프라 설정 |
| 컨텍스트 | GEMINI.md + Knowledge Base (KI 아티팩트, 영속) |
| 워크플로우 | `/start`, `/end`, `/organize`, `/qa-qc` 내장 |
| 제약 | 코드 실행 환경 제한적 (로컬 터미널 접근 방식 다름) |
| 최적 태스크 | 파이프라인 구현/운영, Task Scheduler 관리, QC 세션, 설정 파일 관리 |

### Codex (OpenAI)

| 항목 | 내용 |
|------|------|
| 강점 | 빠른 코드 생성, 단일 파일 집중 수정, 프론트엔드 작업 |
| 컨텍스트 | AGENTS.md (세션 단위, 영속 없음) |
| 제약 | 멀티파일 컨텍스트 약함, 장기 상태 유지 불가 |
| 최적 태스크 | 단일 컴포넌트 구현, 프론트엔드 UI, 빠른 프로토타이핑, 현황 점검 |

## 태스크 유형별 추천 도구

| 태스크 유형 | 1순위 | 2순위 | 비고 |
|-------------|-------|-------|------|
| 크리티컬 버그 수정 | Claude | Gemini | 멀티파일 추적 필요 |
| 신규 기능 구현 (백엔드) | Claude | Gemini | 아키텍처 이해 필요 |
| 신규 기능 구현 (프론트엔드) | Codex | Claude | 컴포넌트 단위 작업 |
| 파이프라인 운영/모니터링 | Gemini | Claude | 장시간 세션에 유리 |
| 테스트 작성/커버리지 | Claude | Codex | QA 전문성 |
| QA/QC 검증 | Gemini | Claude | 체계적 검증 워크플로우 |
| 리팩토링 (대규모) | Claude | Gemini | 넓은 컨텍스트 윈도우 |
| 설정/인프라 관리 | Gemini | Claude | Task Scheduler 등 |
| 코드 리뷰/감사 | Claude | Gemini | 보안/품질 분석 |
| 현황 점검/탐색 | Codex | any | 빠른 스캔 |
| 문서화 | any | - | 작업한 도구가 직접 |

## 협업 패턴

### 릴레이 패턴 (가장 일반적)
1. 도구 A가 작업 → HANDOFF.md 갱신 → TASKS.md 업데이트
2. 도구 B가 HANDOFF.md 읽고 이어서 작업

### 분업 패턴
1. 사용자가 TASKS.md에 담당 도구 지정
2. 각 도구가 자기 담당 태스크만 처리

### 검증 패턴
1. 도구 A가 구현 → TASKS.md에 "QC 필요" 태스크 추가
2. 도구 B(보통 다른 도구)가 QC 수행 → DONE으로 이동
