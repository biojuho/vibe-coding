# 📋 세션 워크플로우 SOP

> 1계층 지침: 에이전트가 매 작업 세션의 시작, 진행, 종료 시 따라야 하는 표준 운영 절차.

---

## 🟢 세션 시작 절차

### 입력

- 사용자의 오늘 작업 설명 (선택)

### 절차

1. **Knowledge Base 확인**
   - `C:\Users\박주호\.gemini\antigravity\knowledge\` 하위의 관련 KI를 확인한다.
   - 특히 `vibe_coding_workspace` KI의 최신 상태를 읽는다.

2. **이전 작업 히스토리 확인**
   - 최근 대화 이력에서 마지막 세션의 TODO를 파악한다.
   - 미완성 작업이나 알려진 버그를 식별한다.

3. **현재 상태 요약 보고**
   - 사용자에게 다음을 보고한다:
     - 마지막 세션 요약
     - 미완성 TODO 항목
     - 알려진 이슈/버그

4. **오늘 작업 계획 수립**
   - 사용자의 요청을 기반으로 Implementation Plan을 세운다.
   - `.agents/rules/project-rules.md`의 규칙을 준수하며 계획한다.

### 사용자 프롬프트 예시

```text
Knowledge Base에서 이전 작업 기록을 확인해줘.
현재 프로젝트 상태와 남은 TODO를 요약한 뒤,
오늘 작업할 내용: [설명]에 대한 Implementation Plan을 세워줘.
```

---

## 🔵 작업 중 절차

### 규칙

1. **Rules 준수**: `.agents/rules/project-rules.md`의 폴더 구조와 코딩 컨벤션을 반드시 따른다.
2. **도구 먼저 확인**: 새 스크립트 작성 전 `execution/` 폴더에 기존 도구가 있는지 확인한다.
3. **지침 확인**: 반복 작업은 `directives/` 폴더의 SOP를 우선 참고한다.
4. **자가 수정**: 오류 발생 시 수정 → 테스트 → 지침 업데이트 루프를 따른다.

### 새 기능 추가 시 프롬프트 예시

```text
[기능 설명]을 구현해줘.
Rules의 폴더 구조와 코딩 컨벤션을 반드시 따르고,
완료 후 변경사항을 Knowledge Base에 기록해줘.
```

---

## 🔴 세션 종료 절차

### 종료 시 입력

- 에이전트의 작업 결과

### 종료 절차

1. **변경사항 기록**
   - 변경된 파일 목록과 변경 이유를 정리
   - 주요 의사결정과 그 근거를 기록

2. **미완성/버그 이슈 정리**
   - 현재 알려진 버그나 문제점 목록화
   - 각 이슈의 심각도와 우선순위 표기

3. **다음 세션 TODO 작성**
   - 이어서 할 작업 항목을 우선순위별로 정리
   - 예상 작업량과 의존성 명시

4. **Knowledge Base 업데이트**
   - 위 1~3번 내용을 Knowledge Base에 저장
   - 관련 KI가 있으면 업데이트, 없으면 새로 생성

### 사용자 프롬프트 예시

```text
오늘 작업을 마무리하면서:
1. 변경된 파일과 작업 내용을 Knowledge Base에 저장
2. 현재 미완성/버그 이슈 정리
3. 다음 세션에서 이어할 TODO 목록 작성
```

## 도구

| 도구 | 용도 |
| ---- | ---- |
| Knowledge Base (KI) | 컨텍스트 저장 및 조회 |
| `task.md` artifact | 세션 내 진행 상황 추적 |
| `implementation_plan.md` | 작업 계획 문서화 |
| `walkthrough.md` | 완료된 작업 증거 문서화 |

## 예외 처리

- **Knowledge Base 접근 불가**: 대화 이력에서 직접 컨텍스트를 복구한다.
- **이전 세션 정보 없음**: 프로젝트 파일 시스템을 직접 스캔하여 현재 상태를 파악한다.
- **급한 핫픽스**: 세션 시작 절차를 생략하고 즉시 수정 후, 종료 절차에서 기록한다.
