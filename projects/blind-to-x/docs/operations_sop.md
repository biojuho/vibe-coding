# Blind-to-X 운영 SOP

> 목적: 운영자가 Notion에서 무엇을 보고 판단해야 하는지 헷갈리지 않게, 검토 중심으로 하루 운영 흐름을 고정합니다.

## 1. 오전 체크

### 09:00

1. Notion `검토 워크플로우` 보드에서 `검토필요` 카드 수를 확인합니다.
2. `검토 큐` 테이블에서 아래 컬럼만 먼저 봅니다.
   - `운영자 해석`
   - `근거 앵커`
   - `검토 포인트`
   - `피드백 요청`
   - `위험 신호`
3. 에러 로그가 필요하면 최근 로그만 확인합니다.

```powershell
cd blind-to-x
Get-Content logs\pipeline.log | Select-String "ERROR" | Select-Object -Last 10
```

## 2. 콘텐츠 검토

### 10:00 ~ 11:00

1. `검토필요` 카드 하나를 엽니다.
2. 본문보다 먼저 `검토 포인트`와 `피드백 요청`을 읽습니다.
3. 문제가 있으면 자유서술보다 먼저 `반려 사유`를 체크합니다.
4. 과한 경고라고 느껴지면 `위험 신호`를 줄이거나 비웁니다.
5. 최종 상태를 아래 중 하나로 바꿉니다.
   - `승인됨`
   - `보류`
   - `반려`

### 빠른 판단 기준

- `운영자 해석`이 과장 없이 납득되는가
- `근거 앵커`가 실제 초안에 살아 있는가
- `검토 포인트`가 초안과 어긋나지 않는가
- `위험 신호`가 진짜 문제인가, 과한 경고인가
- 반려 시 이유가 `반려 사유`로 남아 있는가

## 3. 발행

### 11:00 ~ 12:00

1. `승인됨` 상태만 확인합니다.
2. `발행 플랫폼` 기준으로 필요한 채널의 초안을 복사합니다.
3. 실제 발행은 사람이 수동으로 진행합니다.
4. 발행 후 아래 속성을 채웁니다.
   - `발행 플랫폼`
   - `발행 시각`
   - 링크 속성들
   - `상태 = 발행완료`

> 기본 운영은 자동 발행이 아니라 수동 검토, 수동 발행입니다.

## 4. 필수 명령어

### Notion 스키마 점검

```powershell
py -3 scripts/notion_doctor.py --config config.yaml
```

### 리뷰 컬럼 동기화

```powershell
py -3 scripts/sync_notion_review_schema.py --config config.yaml --apply
```

### 기존 카드 리뷰 컬럼 백필

```powershell
py -3 scripts/backfill_notion_review_columns.py --config config.yaml --apply
```

## 5. 에러 대응

| 증상 | 먼저 볼 것 | 대응 |
|---|---|---|
| Notion 업로드 실패 | `NOTION_DATABASE_ID`, 공유 권한 | `scripts/notion_doctor.py` 실행 |
| 검토 컬럼이 안 보임 | 스키마 sync 여부 | `scripts/sync_notion_review_schema.py --apply` |
| 예전 카드가 비어 보임 | 백필 여부 | `scripts/backfill_notion_review_columns.py --apply` |
| 초안 품질이 반복적으로 어색함 | `반려 사유`, `위험 신호` 누적 | `feedback_loop.py` 기반 운영 메모 확인 |

## 6. 회고 포인트

주 1회 아래만 봅니다.

- 어떤 `반려 사유`가 반복되는가
- 어떤 `위험 신호`가 실제론 과한 경고였는가
- 특정 `토픽 클러스터 + 훅 타입` 조합이 자주 잘리는가
- 승인률이 높아도 운영자가 계속 손대는 영역이 있는가

이 데이터는 다음 초안 프롬프트의 운영 메모로 다시 들어갑니다.
