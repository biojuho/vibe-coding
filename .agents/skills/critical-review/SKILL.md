# Critical Review Skill — 7단계 비판적 검증 프로토콜

## Purpose

외부 정보를 무비판적으로 수용하는 것을 방지한다. 모든 외부 정보는 7단계 검증을 거쳐야 프로젝트에 반영된다.

## When to Use

- 사용자가 500자 이상의 외부 텍스트를 붙여넣을 때
- 사용자가 URL과 함께 "분석해줘", "검토해줘", "이거 어때" 등을 요청할 때
- `/intake` 슬래시 커맨드 호출 시

## When NOT to Use

- 공식 기술 문서(API docs, RFC, SDK reference) → 바로 활용
- 사용자가 "검토 없이 요약만"이라고 명시한 경우
- 코드 리뷰 → `code-reviewer` 에이전트가 담당

---

## 7-Phase Process

### Phase 1: 출처 검증 (Source Verification)

**질문:** 원저자는 누구인가? 1차 자료인가, 2차 해석인가?

| 확인 항목 | 세부 |
| --------- | ---- |
| 자료 유형 | 1차(원본 연구·데이터·공식 발표) vs 2차(재해석·요약) vs 3차(재가공) vs 출처 불명 |
| 원저자 | 이름, 소속, 전문 분야 |
| 원본 링크 | 확인 가능 여부 |
| 왜곡 여부 | 2차 해석일 경우, 원문 대비 누락·왜곡·과장 지적 |

**출력:**

```text
[출처 등급] 1차 자료 / 2차 해석 / 3차 재가공 / 출처 불명
[원저자]    이름, 소속, 전문 분야
[원본 링크] 확인 가능 여부
```

### Phase 2: 이해충돌 검증 (Conflict of Interest Check)

**질문:** 왜 이 사람이 이 말을 하는가?

| 확인 항목 | 세부 |
| --------- | ---- |
| 금전적 이익 | 해당 주장으로 금전적 이익을 얻는가 |
| 이해관계 | 스폰서, 광고주, 소속 기업과의 관계 |
| 홍보 동기 | 특정 제품·서비스·이념 홍보 의도 |

**출력:**

```text
[이해충돌] 없음 / 잠재적 / 명백함
[근거]      구체적 이유 1줄 요약
```

### Phase 3: 근거 검증 (Evidence Verification)

**질문:** 각 핵심 주장의 근거는 무엇인가?

| 근거 등급 | 정의 | 신뢰도 |
| --------- | ---- | ------ |
| **A — 실증 데이터** | 재현 가능한 실험, 통계, 계량 데이터 | ★★★★★ |
| **B — 사례 증거** | 특정 기업·인물의 성공/실패 사례 | ★★★☆☆ |
| **C — 전문가 의견** | 해당 분야 전문가의 판단 (데이터 없이) | ★★★☆☆ |
| **D — 일반 의견** | 비전문가의 주장, 상식적 추론 | ★★☆☆☆ |
| **E — 추측/예측** | 근거 없는 미래 전망, 감정 기반 판단 | ★☆☆☆☆ |

추가 확인:
- 데이터 → 표본 크기, 방법론, p-value/신뢰구간 명시 여부
- 사례 → 생존자 편향 여부
- "연구에 따르면" → 어떤 연구인지 특정 가능 여부

**출력:**

```text
[주장 1] "핵심 주장 요약"
  → 근거 등급: A/B/C/D/E
  → 구체적 근거: 있음(출처) / 모호함 / 없음
```

### Phase 4: 논리 구조 검증 (Logic & Fallacy Check)

**질문:** 근거와 결론 사이의 논리적 연결이 유효한가?

점검할 논리 오류 체크리스트:

- [ ] **인과 오류** — 상관관계를 인과관계로 오인
- [ ] **생존자 편향** — 실패 사례 무시, 성공 사례만 근거 제시
- [ ] **확증 편향** — 자기 주장 뒷받침 증거만 선택적 인용
- [ ] **권위 호소** — "유명한 사람이 말했으므로 맞다"
- [ ] **성급한 일반화** — 소수 사례를 전체에 적용
- [ ] **허수아비 논증** — 상대 주장을 왜곡한 뒤 반박
- [ ] **거짓 이분법** — 두 가지 선택지만 제시, 중간 대안 무시
- [ ] **미끄러운 경사면** — 한 단계 변화가 극단적 결과로 이어진다고 주장

**출력:**

```text
[논리 오류] 발견됨 / 발견되지 않음
[유형]      해당 오류 이름
[위치]      텍스트 내 해당 부분 요약
```

### Phase 5: 시간 검증 (Temporal Validity Check)

**질문:** 이 정보는 현재 시점에서도 유효한가?

| 확인 항목 | 세부 |
| --------- | ---- |
| 작성 시점 | YYYY-MM 또는 추정 시기 |
| 업데이트 여부 | 이후 반박 연구나 업데이트 데이터 존재 여부 |
| 분야 변화 속도 | 기술·시장·규제 등 빠르게 변하는 분야인지 |

**출력:**

```text
[작성 시점]  YYYY-MM 또는 추정 시기
[유효성]     현재 유효 / 부분 유효 / 만료 가능성 높음
[근거]       업데이트 필요 사유 1줄
```

### Phase 6: 반대 시나리오 (Counter-Scenario Analysis)

**질문:** 이 결론이 틀릴 수 있는 조건은?

최소 2개의 반대 시나리오를 제시:

```text
만약 _____ 라면, 이 주장은 틀린다.
만약 _____ 라면, 정반대 결론이 나온다.
```

추가 질문:
- 전제 조건 중 현실에서 깨지기 쉬운 것은?
- 다른 분야·시장·규모에서도 동일 적용 가능한가?
- 저자가 의도적으로 생략했을 반례는?

### Phase 7: 적용성 판단 (Applicability Assessment)

**질문:** 건재의 현재 상황에 맞는가?

| 기준 | 판정 옵션 |
| ---- | --------- |
| 업종/규모 적합성 | 맞음 / 부분 적합 / 안 맞음 |
| 전제 조건 | 충족 / 부분 충족 / 미충족 |
| 예상 리스크 | 낮음 / 중간 / 높음 |
| 부분 적용 | 가능 / 불가능 |

**출력:**

```text
[적용 가능성] 높음 / 조건부 / 낮음
[필요 전제]    충족해야 할 조건
[추천 액션]    구체적 다음 단계 1~2개
```

## Final Verdict — 신뢰도 스코어카드

모든 Phase를 종합하여 스코어카드를 출력한다.

```text
┌─────────────────────────────────────────────┐
│           비판적 검증 스코어카드              │
├──────────────────┬──────────────────────────┤
│ 출처 신뢰도      │ ●●●●○ (4/5)             │
│ 이해충돌         │ ●○○○○ (1/5) ⚠️          │
│ 근거 품질        │ ●●●○○ (3/5)             │
│ 논리 건전성      │ ●●●●● (5/5)             │
│ 시간 유효성      │ ●●●○○ (3/5)             │
│ 반대 시나리오    │ N개 제시됨               │
│ 적용 가능성      │ 높음/조건부/낮음          │
├──────────────────┼──────────────────────────┤
│ 종합 판정        │ ✅/⚠️/❌                  │
│                  │ 판정 사유 1줄             │
└──────────────────┴──────────────────────────┘
```

### 종합 판정 기준

| 판정 | 조건 |
| ---- | ---- |
| ✅ **수용** | 모든 항목 3/5 이상, 논리 오류 없음 |
| ⚠️ **조건부 수용** | 1~2개 항목 2/5 이하이거나 경미한 논리 오류 |
| ❌ **보류/기각** | 3개 이상 항목 2/5 이하이거나 심각한 논리 오류 |

---

## 저장 규칙

- **✅ 수용** → 분야별 폴더에 인사이트 저장 + 원문 링크 첨부
- **⚠️ 조건부 수용** → `_review/` 폴더에 저장, 추가 검증 필요 태그 부착
- **❌ 보류/기각** → 저장하지 않음. 기각 사유만 1줄 기록

### 저장 형식

```text
## [YYYY-MM-DD] 인사이트 제목
- 출처: 원문 링크
- 판정: ✅/⚠️/❌
- 핵심 인사이트: 1~2문장 요약
- 적용 액션: 구체적 행동 1~2개
- 주의사항: 검증에서 발견된 약점
```

## Cross-Tool Sync

이 스킬은 Claude Code의 `.claude/commands/intake/index.md`와 동일한 7단계 프로세스를 정의한다.
어느 AI 도구에서든 동일한 검증 기준이 적용되도록 변경 시 양쪽을 함께 업데이트할 것.
