# Audience-First Skill

> **핵심 원칙**: 산출물을 만들기 전에 항상 "누구를 위한 것인가"를 먼저 확인한다.
> 기술적으로 완벽한 코드도, 문법적으로 완벽한 글도, 타깃 청중에게 맞지 않으면 실패한다.

---

## 1. 언제 이 스킬을 쓰는가

다음 중 하나라도 해당하면 청중 확인 단계를 먼저 진행한다:

| 요청 유형 | 트리거 예시 |
|-----------|------------|
| **코딩/앱** | "앱 만들어줘", "웹사이트", "대시보드", "서비스 개발" |
| **콘텐츠** | "블로그 글", "뉴스레터", "영상 스크립트", "소개글" |
| **디자인** | "랜딩페이지", "포스터", "UI 디자인", "슬라이드" |
| **기획** | "서비스 기획", "제품 로드맵", "출시 전략" |
| **배포/마케팅** | "배포", "런칭", "홍보", "포지셔닝" |

**예외** (청중 확인 생략해도 되는 경우):
- 명확히 개인 유틸리티 ("내 할 일 목록", "내가 쓸 스크립트")
- 이미 청중이 대화에서 충분히 정의된 경우
- 단순 코드 수정/버그 픽스

## 2. 청중 파악 프레임워크

### Step 1: 배포 범위 먼저 확인

```
이 산출물은:
  A) 외부 유저/고객에게 배포되는 것인가?  →  [외부용] 워크플로우
  B) 팀 내부에서만 쓰는 것인가?           →  [내부용] 워크플로우
  C) 콘텐츠 (글/영상/발표자료)인가?        →  [콘텐츠] 워크플로우
```

### Step 2: 유형별 필수 확인 항목

각 워크플로우의 상세 내용은 `references/` 참조.
빠른 요약:

**[외부용 제품/서비스]** → `references/external-product.md`
- 타깃 유저 페르소나 (연령, 직군, 기술 수준)
- 핵심 문제/페인포인트
- 경쟁 대안 대비 차별점
- 유저가 기대하는 핵심 경험 (Core UX)
- 비즈니스 모델 (무료/유료/내부 도구화)

**[내부용 도구]** → `references/internal-tool.md`
- 사용자 (팀원 규모, 기술 수준)
- 현재 수동 프로세스와 Pain point
- 성공 지표 (시간 절약, 오류 감소 등)
- 유지보수 주체

**[콘텐츠 — 글/영상/발표]** → `references/content.md`
- 주요 독자/시청자 페르소나
- 독자의 현재 지식 수준
- 콘텐츠를 접하는 채널/맥락
- 독자가 콘텐츠 후 취해야 할 행동 (CTA)
- 원하는 감정적 반응

---

## 3. 실행 절차

### Phase 0: 청중 인터뷰 (항상 먼저)

요청을 받으면 **바로 만들지 말고**, 아래 질문 중 모르는 것을 먼저 물어본다.
이미 대화에서 답이 나온 항목은 건너뜀.
**한 번에 너무 많이 묻지 말 것** — 가장 중요한 2~3개만 먼저.

```
시작 전 확인 체크리스트:

□ 누가 쓸 것인가? (타깃 유저/독자)
□ 그들의 가장 큰 문제/니즈는?
□ 외부 배포용인가, 내부용인가?
□ 이게 성공하면 어떤 모습인가? (성공 기준)
□ 비슷한 경쟁/대안이 있는가?
```

### Phase 1: 청중 프로필 작성

인터뷰 후 산출물 시작 전에 간략한 청중 프로필을 먼저 제시:

```markdown
## 청중 프로필 (확인 후 진행)

**타깃**: [페르소나 설명]
**핵심 니즈**: [그들이 진짜 원하는 것]
**배포 범위**: [외부/내부/콘텐츠]
**성공 기준**: [이게 잘 됐다는 기준]
**설계 방향**: [이를 바탕으로 어떻게 만들 것인지 한 줄 방향]
```

유저가 OK하면 그때 실제 산출물 제작 시작.

### Phase 2: 청중 중심 설계 반영

산출물 제작 시 청중 프로필을 계속 참고:

- **코드/UI**: UX 결정 시 타깃 기술 수준 반영 (비기술자 → 단순화, 개발자 → 파워유저 기능)
- **콘텐츠**: 어조(tone), 전문 용어 수준, 예시 선택 모두 독자 기준으로
- **구조**: 타깃이 원하는 정보를 먼저, 부가 정보는 뒤로

### Phase 3: 마케팅 체크 (외부 배포용일 때)

외부용 제품/콘텐츠 완성 후 선택적으로 제공:

```
마케팅 셀프 체크
- 타깃이 처음 5초 안에 "이게 나를 위한 것"임을 알 수 있는가?
- 핵심 가치 제안(Value Proposition)이 명확한가?
- CTA(다음 행동 유도)가 있는가?
- 경쟁 대안 대비 차별점이 드러나는가?
```

---

## 4. 산출물 유형별 빠른 가이드

### 코딩/앱 요청
1. 청중 프로필 확인
2. 기능 목록 작성 전에 "유저 스토리" 먼저:
   `"[타깃 유저]로서, [목표]를 달성하기 위해 [기능]이 필요하다"`
3. MVP 범위 제안 (처음부터 다 만들지 않기)
4. 외부 배포용이면 온보딩/빈 상태(Empty State) 설계 포함

### 블로그/뉴스레터/글
1. 독자 페르소나 + 읽는 맥락 확인
2. 제목 옵션 2~3개 먼저 제안 (클릭 유도)
3. 글 구조: 훅(Hook) → 문제 → 해결 → CTA
4. 전문 용어 수준 조정

### 영상 스크립트
1. 시청자 페르소나 + 시청 플랫폼 확인
2. 첫 15초 훅 설계 (이탈 방지)
3. 시각 자료 요소 표시
4. 마지막 CTA 명시

### 발표/슬라이드
1. 청중 배경 지식 수준 확인
2. 발표 후 청중이 할 행동/결정 명확히
3. 스토리라인 먼저, 슬라이드는 나중에

## 5. 예시 대화

**스킬 없이 (현재 문제)**
> 유저: "할 일 관리 앱 만들어줘"
> Claude: [바로 To-do 앱 코드 작성...]

**스킬 적용 후**
> 유저: "할 일 관리 앱 만들어줘"
> Claude: "만들기 전에 방향을 잡으면 더 잘 만들 수 있어요.
> - 누가 주로 쓸 예정인가요? (혼자 쓸 개인 도구 vs 팀에 배포)
> - 기존에 쓰던 앱(Todoist, Notion 등)에서 불편했던 점이 있나요?
>
> **청중 프로필**
> 타깃: 1인 창업자, 30대, 앱에 익숙함
> 핵심 니즈: 빠른 입력 + 우선순위 관리
> 배포 범위: 개인용 (내부)
> 성공 기준: 매일 열어서 씀
> 설계 방향: 입력 마찰 최소화, 복잡한 기능 배제
>
> 이 방향으로 만들까요?"
