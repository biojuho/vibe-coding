# /qa-qc — Vibe Coding QA/QC 워크플로우

// turbo-all

## 개요

코드 품질 보증을 위한 4단계 파이프라인.
각 STEP은 독립 노드로 실행 가능하며, 마스터 프롬프트로 한 번에 통합 실행도 가능합니다.

| STEP | 역할 | 입력 | 출력 |
|------|------|------|------|
| 1 | 개발 | `user_request` | 코드 + 테스트 |
| 2 | QA 검토 | STEP 1 출력 | QA 결과표 |
| 3 | 수정 | STEP 1 + 2 출력 | 최종 코드 |
| 4 | QC 보고서 | STEP 3 출력 + 원래 요청 | QC 보고서 |

---

## 실행 절차

### STEP 1 — 개발

시니어 소프트웨어 엔지니어로서 코드를 작성한다.

**규칙:**
- 가독성 높은 코드, 명확한 함수/변수명
- 모든 함수에 간단한 주석 포함
- 환경변수/API 키/시크릿 하드코딩 금지
- 엣지 케이스(빈 값, null, 예외) 반드시 고려
- 유닛 테스트 코드 함께 작성

**출력:** 구현 코드 → 테스트 코드 → 사용 방법 3줄 요약

---

### STEP 2 — QA 자동 검토

날카로운 QA 엔지니어로서 STEP 1 결과를 검토한다.

**검토 항목:**

1. **기능성**: 요구사항 충족, 누락 기능, 입력값 유효성
2. **보안**: SQL Injection/XSS/CSRF, 민감 정보 노출, 외부 입력 신뢰 처리
3. **안정성**: 예외 처리 충분성, 무한 루프/메모리 누수, 비동기 오류
4. **코드 품질**: DRY 원칙, SRP, 테스트 코드 검증 범위

**출력:** QA 결과 (PASS ✅ / FAIL ❌) + 문제 목록 테이블 + 총평

---

### STEP 3 — 코드 수정

시니어 엔지니어로서 QA 지적사항을 반영하여 코드를 수정한다.

**규칙:**
- QA 지적사항 전항목 빠짐없이 수정
- 미수정 항목은 이유 명시
- 기존 기능 손상 방지
- 수정 부분에 `# [QA 수정]` 주석 태그

**출력:** 최종 코드 → 수정 사항 요약 → 미수정 항목 (있을 경우)

---

### STEP 4 — QC 최종 승인 보고서

QC 매니저로서 배포 전 공식 보고서를 작성한다.

**작성 항목:**
1. 요구사항 충족 여부 (Yes/No + 근거)
2. 최종 체크리스트 (실행 가능, 테스트 포함, 보안, 환경변수, 예외 처리, 문서화)
3. 예상 리스크 (LOW/MED/HIGH)
4. 테스트 시나리오 3가지
5. 롤백 계획
6. 최종 판정: ✅ 승인 / ⚠️ 조건부 승인 / ❌ 반려

---

## 마스터 프롬프트 (통합 실행)

노드 분리 없이 한 번에 실행 시, PHASE 1~4를 순차적으로 처리:

1. **PHASE 1 (개발)**: 구현 + 테스트 + 엣지 케이스
2. **PHASE 2 (QA)**: 기능성/보안/안정성/코드 품질 4가지 기준 검토
3. **PHASE 3 (수정)**: QA 발견 문제 전수 수정 + `# [QA 수정]` 주석
4. **PHASE 4 (QC)**: 체크리스트/리스크/테스트/롤백/최종 판정

---

## 예외 처리

- **STEP 2에서 PASS**: STEP 3 생략, 바로 STEP 4 진행
- **STEP 4에서 반려**: STEP 1부터 재시작 (반려 사유 반영)
- **조건부 승인**: 조건 항목만 수정 후 STEP 4 재실행
