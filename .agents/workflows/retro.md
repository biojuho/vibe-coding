# /retro — 주간 개발 회고

> **gstack 철학:** 회고는 잘못을 찾는 게 아니라, 다음 주를 더 잘하기 위한 것이다.
> 데이터 기반으로 패턴을 찾고, 시스템을 개선한다.

// turbo-all

## 언제 사용하나?

- 매주 금요일 또는 스프린트 종료 시
- "요즘 뭘 하고 있었지?" 할 때
- 기술 부채가 쌓이는 것 같을 때

---

## 실행 절차

### STEP 1 — 지난 7일 작업 통계 수집

```bash
# 지난 7일 커밋 수 확인
git log --since="7 days ago" --oneline --all

# 변경된 파일 통계
git log --since="7 days ago" --stat --all | tail -20
```

위 명령어를 실행하고 결과를 파싱하여 다음을 정리한다:
- 총 커밋 수
- 주요 변경 파일 목록 (상위 5개)
- 프로젝트별 작업 분포

---

### STEP 2 — Knowledge Base에서 세션 기록 확인

최근 대화 이력(conversation summaries)과 `.ai/SESSION_LOG.md`에서:
- 완료된 작업 항목 목록
- 미완료/진행 중 항목
- 발견된 버그 중 미해결 건

---

### STEP 3 — 기술 부채 현황 파악

각 프로젝트에서 다음을 확인한다:

| 점검 항목 | 방법 |
|-----------|------|
| TODO/FIXME 주석 수 | `grep -r "TODO\|FIXME" --include="*.py" --include="*.ts"` |
| 테스트 커버리지 | 각 프로젝트의 pytest/jest 리포트 확인 |
| 오래된 의존성 | `pip list --outdated` / `npm outdated` |
| 미해결 이슈 | GitHub Issues 또는 `.tmp/` 내 이슈 파일 |

---

### STEP 4 — 회고 보고서 작성

다음 형식으로 보고서를 작성한다:

```markdown
## 📊 주간 회고 — YYYY년 MM월 W주차

### 이번 주 한 일
- ✅ [완료 항목 1]
- ✅ [완료 항목 2]

### 미완료 항목
- 🔄 [진행 중] [항목] — 이유: ...
- ⏸️ [보류] [항목] — 이유: ...

### 기술 부채 현황
- 🔴 높음: [건수] 건
- 🟡 보통: [건수] 건
- 🟢 낮음: [건수] 건

### 잘한 것 (Keep)
- ...

### 개선할 것 (Improve)
- ...

### 다음 주 우선순위 (Top 3)
1. [최우선]
2. [두 번째]
3. [세 번째]
```

---

### STEP 5 — Knowledge Base 업데이트

보고서를 Knowledge Base에 저장한다 (`/end` 워크플로우의 STEP 4와 동일한 방식).

---

## 출력

1. **주간 회고 보고서** (위 형식)
2. **다음 주 우선순위 Top 3**
3. **기술 부채 해소 제안** (있을 경우)

---

## 예외 처리

- **첫 회고 실행**: 지난 7일 데이터가 없으면 가능한 범위에서 수집
- **커밋 없음**: 코드 외 작업(기획, 회의, 학습) 중심으로 정리
