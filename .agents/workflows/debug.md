# /debug — 체계적 근본 원인 디버깅

> **gstack Iron Law: 조사 없이 수정하지 않는다.**
> 증상을 보고 바로 코드를 바꾸는 건 의사가 증상만 보고 수술하는 것과 같다.
> 반드시 원인을 찾고, 가설을 세우고, 검증한 뒤에 수정한다.

// turbo-all

## 언제 사용하나?

- 버그가 발생했는데 원인을 모를 때
- 수정했는데 또 같은 문제가 발생할 때
- 에러 메시지는 있는데 어디서 왜 나는지 파악이 안 될 때

---

## 실행 절차

### STEP 0 — 증상 기록 (수정 금지)

먼저 **현상을 정확히 기록**한다. 코드는 건드리지 않는다.

```
증상: [에러 메시지 / 잘못된 동작 설명]
발생 조건: [어떤 상황에서 발생하나?]
재현 방법: [몇 번에 한 번? 항상? 특정 입력에서만?]
최근 변경: [최근에 뭘 바꿨나? 언제부터 발생했나?]
```

---

### STEP 1 — 데이터 흐름 추적

버그가 발생하는 경로를 입력에서 출력까지 추적한다.

**방법:**
1. 에러 스택 트레이스를 top-down으로 읽는다
2. 각 함수/모듈 경계에서 데이터가 어떻게 변하는지 확인한다
3. 가장 먼저 이상한 값이 나타나는 지점을 찾는다

```python
# Python 디버깅 패턴
import logging
logging.basicConfig(level=logging.DEBUG)
# 또는
print(f"[DEBUG] {variable_name=}")  # Python 3.8+ f-string debug
```

```typescript
// TypeScript 디버깅 패턴
console.log('[DEBUG]', { variableName });
```

---

### STEP 2 — 근본 원인 가설 수립

데이터 추적 결과를 바탕으로 **최대 3가지 가설**을 세운다.

| 가설 번호 | 가설 내용 | 검증 방법 | 예상 증거 |
|-----------|-----------|-----------|-----------|
| H1 | ... | ... | ... |
| H2 | ... | ... | ... |
| H3 | ... | ... | ... |

가장 가능성 높은 가설부터 검증한다.

---

### STEP 3 — 가설 검증 (최대 3회 시도)

각 가설을 검증한다. **규칙:**

- 한 번에 **하나의 변경만** 적용한다 (변수 격리)
- 변경 전후 동작을 비교한다
- 검증에 실패하면 변경을 **롤백**하고 다음 가설로 넘어간다

**시도 1:**
```
가설: H1
변경 내용: ...
결과: ✅ 해결 / ❌ 실패 → 롤백
```

**시도 2:**
```
가설: H2
변경 내용: ...
결과: ✅ 해결 / ❌ 실패 → 롤백
```

**시도 3:**
```
가설: H3
변경 내용: ...
결과: ✅ 해결 / ❌ 실패 → 아래 예외 처리
```

> ⚠️ **3회 연속 실패 시 중지.** 사용자에게 현황을 보고하고 새로운 정보를 요청한다.
> 추측으로 계속 시도하는 것은 시간 낭비이며 코드를 더 망가뜨릴 수 있다.

---

### STEP 4 — 수정 확인 및 회귀 테스트 생성

버그 수정 성공 후:

1. **재현 테스트** 작성: 이 버그를 다시 잡기 위한 테스트
   ```python
   def test_regression_[버그_설명]():
       # 버그를 재현하는 입력
       # 수정 후 기대 동작 검증
   ```

2. **엣지 케이스 추가**: 비슷한 패턴의 다른 입력도 테스트

3. **`/qa-qc` STEP 4** 기준으로 최종 확인

---

### STEP 5 — 사후 분석 (Post-mortem)

버그 해결 후 한 줄 요약을 기록한다:

```
날짜: YYYY-MM-DD
버그: [한 줄 설명]
근본 원인: [왜 발생했나?]
수정 방법: [어떻게 고쳤나?]
재발 방지: [어떻게 막을 건가?]
```

이 내용을 `.ai/SESSION_LOG.md`에 추가한다.

---

## 출력

1. **근본 원인 분석 결과**
2. **수정된 코드**
3. **회귀 테스트**
4. **Post-mortem 한 줄 요약**

---

## 예외 처리

- **3회 실패 시**: 사용자에게 다음을 보고한다
  - 시도한 가설 목록
  - 각 시도 결과
  - 현재 알고 있는 정보
  - 추가로 필요한 정보 또는 로그

- **환경 차이**: "내 컴퓨터에서는 됨" → Windows/Linux 경로차이, 환경변수, 인코딩 먼저 확인
- **비결정적 버그** (간헐적 발생): 로그를 먼저 충분히 수집한 뒤 접근
  - Python: `logging.basicConfig(level=logging.DEBUG)` 후 `.tmp/debug_[날짜].log`에 출력 저장
  - 재현 가능한 최소 입력값을 찾는 것이 최우선
  - 최소 3회 이상 발생 로그 확보 후 패턴 분석
